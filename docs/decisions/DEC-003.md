# Keep the server as the authoritative consistency boundary

## Context and Problem Statement

Promethea clients need to browse and modify catalog data, reading state, metadata, assets, and future sync/job state. The SRS states that clients should communicate changes through the server API or an approved local embedded-backend interface, not by directly editing the server database file.

## Decision Drivers

* [REQ-INT-008](./../requirements/interface/REQ-INT-008.md) requires client database isolation.
* [REQ-FUNC-027](./../requirements/functional/REQ-FUNC-027.md) requires remote updates through the server.
* [REQ-REL-001](./../requirements/reliability/REQ-REL-001.md) requires transactional catalog mutations.
* [REQ-MAINT-002](./../requirements/maintainability/REQ-MAINT-002.md) requires database migrations to be controlled by the backend.
* The final database engine remains open, so clients must not depend on database implementation details.

## Considered Options

* Server-authoritative REST/API boundary
* Allow clients to directly read and modify the database
* Client-local replicated databases with synchronization

## Decision Outcome

Chosen option: "Server-authoritative REST/API boundary", because it protects consistency, keeps persistence details behind the backend, and supports future changes to the database engine or deployment topology.

### Consequences

* Good, because all catalog mutations can be validated, authorized, logged, and migrated through one boundary.
* Good, because browser, desktop, and mobile-oriented clients can share the same API contract.
* Good, because the database engine can change without breaking clients.
* Bad, because offline-first clients are not supported by default.
* Bad, because local desktop workflows require either a local server/embedded backend or a separately approved design.

### Confirmation

Confirm by reviewing client data-access code. Clients must not open or modify the server database file directly. Catalog mutation tests should exercise REST/API paths, and migrations should be owned by the backend. Any local embedded-backend exception must be documented by a later ADR.

## Pros and Cons of the Options

### Server-authoritative REST/API boundary

* Good, because it satisfies client database isolation and remote update requirements.
* Good, because it centralizes validation, authorization, and transactional behavior.
* Good, because it enables later database-engine replacement.
* Neutral, because an embedded local backend may still be allowed if it preserves the same boundary.
* Bad, because it makes the server/API mandatory for write workflows.

### Allow clients to directly read and modify the database

* Good, because some local workflows may appear simpler initially.
* Bad, because it creates a high risk of corruption, migration conflicts, and inconsistent validation.
* Bad, because it conflicts with the SRS database isolation requirement.
* Bad, because it couples clients to the database schema.

### Client-local replicated databases with synchronization

* Good, because it could support offline-first and multi-device local editing.
* Neutral, because it may become relevant in a future sync-heavy architecture.
* Bad, because conflict resolution and replication would add major complexity.
* Bad, because it is disproportionate for the initial personal/self-hosted scope.
