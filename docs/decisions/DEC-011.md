# Use hexagonal architecture in Rust backend

## Context and Problem Statement

In complex applications, it is vital to have a rough plan for the architecture. This avoids messy code with no clear structure that will be much harder to fix in the future. 

<!-- This is an optional element. Feel free to remove. -->
## Decision Drivers

* [REQ-BUILD-001](./../requirements/build/REQ-BUILD-001.md) requires a Rust backend
* [REQ-MAINT-001](./../requirements/maintainability/REQ-MAINT-001.md) requires a modular backend
* [DEC-004](./DEC-004.md) defines the use of adapters

## Considered Options

* Hexagonal backend
* Micro-service backend
* Monolithic backend

## Decision Outcome

Chosen option: "Hexagonal backend", because it was already the chosen architecture in the [`pre-alpha`](https://github.com/praktikantonelli/promethea/tree/pre-alpha) stage, and it seems the ideal middle-ground between monolithic and micro-service.

### Consequences

* Good, because different parts of the system stay independent
* Good, because dependency flow is easy to follow
* Good, because different hosts can use different adapters
* Bad, because it needs more effort to set up

## Pros and Cons of the Options

### Hexagonal backend

* Good, because different parts of the system stay independent
* Good, because dependency flow is easy to follow
* Good, because different hosts can use different adapters
* Bad, because it needs more effort to set up

### Micro-service backend

* Good, because different parts of the system stay independent
* Neutral, because communication between different features is universal
* Neutral, because it is not sensible for small scales
* Bad, because it requires a lot of developmental effort

### Monolithic backend

* Good, because development is very fast
* Good, because there are fewer indirections in implementations, interfaces etc.
* Bad, because maintainability suffers via tight coupling
