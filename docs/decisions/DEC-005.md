# Store catalog data separately from managed binary assets

## Context and Problem Statement

Promethea must persist structured catalog records as well as binary assets such as EPUBs, cover images, author images, imported originals, generated thumbnails, backups, and previous EPUB versions. These two kinds of data have different access patterns, consistency needs, growth behavior, backup concerns, and query requirements.

## Decision Drivers

* [REQ-FUNC-002](./../requirements/functional/REQ-FUNC-002.md) requires persistence of catalog records and assets.
* [REQ-INST-002](./../requirements/installation/REQ-INST-002.md) requires a configurable data directory.
* [REQ-PERF-003](./../requirements/performance/REQ-PERF-003.md) requires asset storage growth visibility.
* [REQ-AVAIL-002](./../requirements/availability/REQ-AVAIL-002.md) requires backup and restore support.
* EPUB mutation/versioning requires preserving binary file versions.

## Considered Options

* Database for structured catalog data plus managed asset storage
* Store all binary assets inside the database
* Store only files on disk and infer catalog state from filenames or EPUB metadata

## Decision Outcome

Chosen option: "Database for structured catalog data plus managed asset storage", because it keeps catalog state queryable and transactional while allowing efficient lifecycle management for large binary files.

### Consequences

* Good, because catalog queries, filters, analytics, and migrations can use structured persistence.
* Good, because EPUBs and images can be stored, versioned, backed up, and measured as managed assets.
* Good, because large binary files do not bloat the relational catalog store.
* Bad, because the system must maintain referential consistency between catalog records and asset files.
* Bad, because backup/restore must cover both database state and asset storage.

### Confirmation

Confirm by inspecting the persistence model, configured data directory layout, and backup/restore procedure. Database records must reference managed assets through controlled identifiers/paths, and integrity checks should detect missing or orphaned assets.

## Pros and Cons of the Options

### Database for structured catalog data plus managed asset storage

* Good, because it matches the SRS distinction between catalog database and managed assets.
* Good, because it supports metadata queries and reading analytics.
* Good, because it keeps large EPUB/image objects in storage optimized for files.
* Neutral, because the exact storage backend may start as filesystem storage and evolve later.
* Bad, because consistency checks are required across database and asset storage.

### Store all binary assets inside the database

* Good, because database-level transactions can cover catalog and asset bytes together.
* Neutral, because this may be acceptable for small libraries.
* Bad, because database size and backup operations could become heavy as assets grow.
* Bad, because serving and versioning large files can become less efficient.

### Store only files on disk and infer catalog state from filenames or EPUB metadata

* Good, because it minimizes database design upfront.
* Bad, because browsing, search, analytics, reading tracking, and provider metadata need structured state.
* Bad, because it cannot reliably represent user edits, reading state, job history, and relationships.
* Bad, because it would make migrations and consistency difficult.
