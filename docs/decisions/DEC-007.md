# Preserve previous EPUB versions before modifying stored EPUB files

## Context and Problem Statement

Promethea must support writing metadata to EPUB files, editing selected EPUB content, updating cover images, and recovering from failed EPUB modification. These operations can corrupt or undesirably change the user's stored book file if executed in place without a recoverable previous version.

## Decision Drivers

* [REQ-FUNC-005](./../requirements/functional/REQ-FUNC-005.md) requires writing metadata to EPUB files.
* [REQ-FUNC-006](./../requirements/functional/REQ-FUNC-006.md) requires editing EPUB content.
* [REQ-FUNC-007](./../requirements/functional/REQ-FUNC-007.md) requires versioning EPUB modifications.
* [REQ-FUNC-037](./../requirements/functional/REQ-FUNC-037.md) requires updating EPUB cover images.
* [REQ-REL-002](./../requirements/reliability/REQ-REL-002.md) requires recovery from failed EPUB modification.

## Considered Options

* Copy/version the current EPUB before applying mutations
* Modify EPUB files in place without preserving a previous version
* Never modify EPUB files and store only catalog-side overrides

## Decision Outcome

Chosen option: "Copy/version the current EPUB before applying mutations", because it provides recoverability and user confidence for destructive or semi-destructive EPUB operations while still supporting real file updates.

### Consequences

* Good, because failed or undesired EPUB mutations can be recovered.
* Good, because version history supports auditability and safer experimentation.
* Good, because metadata and cover updates can be reflected in actual EPUB files.
* Bad, because asset storage grows with each retained version.
* Bad, because retention, cleanup, and restore behavior must be specified.

### Confirmation

Confirm by tests that an EPUB mutation first creates a recoverable previous version and that a failed mutation does not leave the catalog pointing to a corrupted or partially written EPUB. Demonstrations should show restore or rollback behavior for modified EPUB assets.

## Pros and Cons of the Options

### Copy/version the current EPUB before applying mutations

* Good, because it satisfies EPUB versioning and mutation recovery requirements.
* Good, because it allows real EPUB file updates without sacrificing recovery.
* Good, because it can support before/after inspection of modifications.
* Neutral, because retention policy can be refined later.
* Bad, because storage consumption increases with modifications.

### Modify EPUB files in place without preserving a previous version

* Good, because it minimizes storage usage.
* Good, because implementation may be simpler initially.
* Bad, because it risks irreversible corruption or unwanted changes.
* Bad, because it conflicts with versioning and recovery requirements.

### Never modify EPUB files and store only catalog-side overrides

* Good, because original files remain untouched.
* Neutral, because this can be useful as an option for cautious users.
* Bad, because it does not satisfy requirements to write metadata, edit content, or update covers in EPUB files.
* Bad, because exported/downloaded files would not contain the user's desired updates.
